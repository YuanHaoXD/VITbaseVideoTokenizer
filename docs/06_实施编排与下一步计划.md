## §3 审查标准与验收标准

### 3.1 审查标准（review agent 对每张卡做的事）

1. **接口未被改动**：类名/函数签名/张量形状与 05 冻结契约逐字一致（diff 核对）。
2. **编译通过**：`python -m py_compile <file>` 零错误。
3. **验收测试在场且绿**：05 卡"验收"栏指定的测试存在；本机能跑的（不依赖 torch/权重）跑通。
4. **契约一致性**：跨模块的张量/键名/维度对接无误（如 encoder 输出 h 的形状喂得进 GSB）。
5. **ADR 注释**：歧义决策点有 ADR 编号注释，无硬编码魔数。
6. **§6 修订落实**：若该卡涉及已知缺陷修订（S2/normalize 等），确认已按修订实现。

### 3.2 验收标准（acceptance = 合并入主干的条件）

一张卡可合并 ⇔ 同时满足：

- ✅ §3.1 全部 6 条
- ✅ `csv_file=null128` 假数据端到端冒烟通过（涉及训练/前向的卡）
- ✅ review agent 报告无 Critical/Important 级 issue（Minor 记录在案可放行）
- ✅ GLM 5.2（总览）签字：契约未被破坏、与邻接模块对接正确

### 3.3 各卡的硬验收点（05 验收栏 + 本次增量）

| 卡 | 硬验收（必须）|

| M-9 | `test_decompressor_shape`：4 位→16 帧；**输入无锚位**，TemporalUnfold2x 锚帧直通逻辑须规避（见 §6.1 裁决）|

## §4 多 Agent 编排方案（谁做什么、怎么流转）

### 4.1 角色定义

| 角色 | 由谁担任 | 职责 | 工具映射 |
|---|---|---|---|
| **总览/架构** | **GLM 5.2（本机）** | 时序裁决、契约修订、分派、收集审查、签字合并、本文档维护 | `task()` 调度 + 直接编辑 |
| **编码 Agent** | Sisyphus-Junior（按卡选 category）| 单卡实现 + 该卡测试，输入=本卡 05 规格 + §6 修订 + 样板文件 | `task(category=...)` |
| **审查 Agent** | Oracle / Momus（read-only）| 按 §3.1 审查，产出结构化报告 | `task(subagent_type="oracle"/"momus")` |
| **集群执行** | 人（你）/Fable 5 | Linux 上跑 torch 测试 + null 冒烟（本机做不了）| 手动 |

### 4.2 编码 Agent 的输入契约（每次分派必给）

1. 05 §0 全局约定 + 该卡全文
2. §6 涉及该卡的修订条目
3. 两个样板文件（attention_mask.py / temporal_fold.py）作代码风格标尺
4. 该卡依赖的**已落盘模块的真实接口**（不是 05 的预期接口——避免接口漂移踩坑）
5. 明确的"只许动这几个文件"边界

### 4.3 流转（每张卡的生命周期）

```
分派编码(给真实接口) → 编码Agent实现+自测 → py_compile闸 →
审查Agent(§3.1六条) → GLM5.2复核报告 → [通过]签字合并 / [不通过]回炉(带审查意见)
                                                              ↓
                                                    （W1完成后）集群 null 冒烟 → 集成验收
```

### 4.4 依赖序与并行度

```
阶段一(P1解阻,串行关键路径):
  W1(M-10) ──依赖──► W2(M-9,可与W1并行? 否:都属总装但M-9被M-10总装引用,建议先M-10)
```

## §6 契约修订（Fable 5 重点复核项）

> 这些是对 05 冻结接口的**主动偏离/补充**，每条带理由。Fable 5 须确认理由成立、修订无副作用。

### §6.1 【S2 修订】Decompressor 输出维度 & distill head_vid 契约统一

**问题**：05 M-9 规定 Decompressor"末端 Linear 投到 teacher_dim"；但 L-2 的 `head_vid = Linear(student_dim→teacher_vid_dim)` 期望输入是 student_dim。当 teacher_vid_dim ≠ student_dim（InternVideo 几乎必非 1152），head_vid 收到 teacher 宽输入 → **shape mismatch 崩溃**。

**修订（裁决）**：维度投影职责**统一收在 L-2 的对齐头**（与 img_patch/img_pool 三项一致），Decompressor **只做时序上采样、输出 student_dim(D)**。
- M-9 接口改为：`Decompressor(dim, num_heads).forward(s_frames:[B,T1z-1,N,D]) -> [B,4*(T1z-1),N,D]`（删 teacher_dim 参数与末端 Linear）
- L-2 `head_vid = _head(student_dim, teacher_vid_dim)` 现状正确，无需改；删其 docstring 里"末端 Linear 已投教师维度"的过时说明
- **理由**：单一职责 + 三项蒸馏头模式统一 + Stage-3 冻 decoder 训 Sem 侧时投影头归属清晰

**Fable 5 复核点**：① 此修订是否破坏 01 §2.8 的蒸馏语义？② 是否影响 Decompressor 的 checkpoint 不导出逻辑？（答：不影响，仍不导出）

### §6.2 【normalize 持久化】GSB.normalize 不进 state_dict

**问题**（M-3/4/5 交付报告 #4）：`GSB.normalize` 是普通 Python 属性，Stage-3 置 True 后存 checkpoint，重载不自动恢复 → Stage-3 续训/推理时 decoder 误消费未规范化 z。

**修订**：M-10 的 UVTTokenizer 负责——
- `estimate_latent_stats` 末尾把 `self.gsb.normalize=True` 连同 z_mean/z_std（已是 buffer）一起随 checkpoint 存
- 加载钩子：`load_state_dict` 后显式 `model.gsb.normalize = ckpt.get('gsb_normalize', False)`
- **理由**：不改 M-5 接口（契约稳定），把持久化职责放在总装层

### §6.3 【forward→forward_train】DDP 约定（契约②，源自 D/TR 交付报告）

**约定**：DDP 只 hook `forward`。M-10 的 `UVTTokenizer.forward(video)` 在 `self.training=True` 时**必须分派到 `forward_train`**。TR-2 调 `self.model_ddp(x)` 已依赖此约定。**写进 M-10 卡的硬验收**（§3.3）。

### §6.4 【键名别名】video 源 'gt' vs D-2 'video'

D-1（LARP）主键 `'gt'`，D-2 冻结契约是 `'video'`。已裁决：image_dataset 同时返回 `'video'`（守契约）+ `'gt'`（别名，下游零特判）。M-10/TR-2 内部别再假设单一键名。

### §6.5 【GAN 多入口】UVTGANLoss 不进 DDP（契约③）

判别器 + DistillLoss 对齐头是**多入口模块**，不能被 DDP 包（DDP 仅同步 forward 路径）。TR-2 已在窗口末对这些参数手动 all-reduce（等价 DDP mean-reduce，天然免 no_sync）。审查 TR-2 时确认此机制在场。
